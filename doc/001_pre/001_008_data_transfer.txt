================================================================================
現状（Phase 5 済み）と Phase 11 の差
================================================================================

  Phase 5（実装済み・「外へ出すだけ」）:
    - コレクション全件 → JSON ファイル（ダイアログ保存）
    - 画面上のドキュメント一覧（Query 結果等）→ JSON / CSV
    - 向きはエクスポートのみ。Firestore へ書き戻さない

  Phase 11（未着手・本ドキュメント）:
    - ファイル → Firestore への書き戻し（import）
    - プロジェクト丸ごと保全（全 DB バックアップ）
    - 接続済みプロジェクト間のコピー（転送）
    - Emulator とのデータ同期（接続自体は Phase 12）

  一言で言うと:
    Phase 5 = 読取中心のファイル書き出し
    Phase 11 = 書込を含む入出力・保全・接続間コピー（危険度が高い）


================================================================================
目的
================================================================================

  - ローカルファイルと Firestore を往復できるようにする（import で片方向を閉じる）
  - 作業前スナップショット・災害復旧用に「プロジェクト丸ごと」をアーカイブできる
  - workspace に載せた複数プロジェクト間で、コレクション単位のデータを移せる
  - （Phase 12 後）Emulator と実プロジェクトのあいだでも同じ考え方でデータを動かせる

--------------------------------------------------------------------------------
1. Import（CSV / JSON → コレクション）
--------------------------------------------------------------------------------

  やること:
    ローカルファイルを、いま接続中の projectId の指定コレクションへ書き込む。

  ざっくりフロー:
    1. ファイル選択（JSON / CSV）
    2. 宛先: projectId + collectionPath
    3. ドキュメント ID の決め方を指定
    4. プレビュー（件数・サンプル行・衝突数の見積もり）
    5. 実行（batch 書込）→ 結果レポート（成功 / スキップ / 失敗）

  想定入力（Phase 5 export との往復を最優先）:
    JSON:
      ExportDocument[] 相当
        { "id": "...", "path": "...", "data": { ... } }
      ※ path はヒント。書込先の正は「指定コレクション + id」
    CSV:
      ヘッダに id（必須寄り）、任意で path、あとはフィールド列
      Phase 5 の CSV エクスポート（id, path, ...columns）を読めること

  未策定（決めてから実装）:
    □ JSON で「data だけの配列」「単一オブジェクト」など別形をどこまで許すか
        →　id が無い形を許すなら、そのときの ID は SDK 自動採番（下の ID 方針とセット）
           ※ 「別形をどこまで許すか」と「ID の付け方」は別問題。正式形は export 往復用の
             { id, path, data } を本筋にする想定（別形の許否は export と合わせて確定）

    □ ドキュメント ID: ファイルの id 列を使う / 自動生成 / 特定フィールドを ID にする
        →　ファイルに id がある → その id を使う（export 往復の本筋）
           ファイルに id が無い → 自動 ID（firestore 本家 SDK）で書く
           （特定フィールドを ID にする、は v1 ではやらない）

    □ 衝突時: 上書き（set） / スキップ / エラーで全体停止 / マージ（update）
        →　・・エラー停止
           ・・エラーメッセージ

    □ 型変換: CSV は原則すべて string か、数値・bool・null・Timestamp を推定するか
       Timestamp / GeoPoint / DocumentReference の表現は serialize 形式に合わせるか
        →　型もそのまま保存。
           ※ Firestore から読むときは型は取れる。JSON export は __firemint_type で残せる。
             「型が落ちる」のは主に CSV。CSV の推定は export 方針とセットで後決め。

    □ サブコレクション: import 対象外（v1）か、ネスト JSON を許すか
        →　今回、v1では、サブコレクション除外。　


    □ プレビュー必須か、少件数はスキップ可か
        →　プレビュー必須

    □ read-only プロジェクトでは必ず拒否（ensureWritable）
        →　イエス。

    □ production environment のとき追加確認ダイアログを出すか
        →　ＮＯ．　detectEnvironment()をまるごと除外。

2. 全 DB バックアップ（圧縮アーカイブ）
--------------------------------------------------------------------------------

  やること:
    指定 projectId のルートコレクションをたどり、ドキュメント（＋方針次第で
    サブコレクション）をまとめて 1 アーカイブとして保存する。

  Phase 5「1 コレクション JSON」との違い:
    - 対象 = プロジェクト全体（またはユーザーが選んだ複数ルート）
    - 形式 = 複数ファイル + メタ情報を zip（等）に固める
    - 用途 = 保全・作業前スナップショット。画面の「今見えている結果」ではない

  アーカイブイメージ（未確定・例）:

    backup-<projectId>-<timestamp>.zip
      manifest.json   … version, projectId, createdAt, 件数サマリ
      collections/
        users/
          docs.json   … ExportDocument[] 相当
        orders/
          docs.json
      （サブコレクション方針次第で path 階層を再現）

  未策定:
    □ サブコレクションを再帰で含めるか（含めるのが「全 DB」の本義）
    □ Collection Group や除外コレクション指定が要るか
    □ 圧縮形式（zip 固定でよいか）
    □ 巨大プロジェクト時のチャンク・一時ディレクトリ・メモリ上限
    □ 進捗 UI（件数 / コレクション名）の出し方
    □ 「バックアップからの復元（restore）」を Phase 11 に含めるか
       → Done 一覧は export 側のみ。restore は 11-b か別 Phase を推奨


--------------------------------------------------------------------------------
3. プロジェクト間転送（dev ↔ prod 等）
--------------------------------------------------------------------------------

  やること:
    workspace でロード済みの 2 つの projectId 間で、コレクション単位に
    データをコピーする（ファイルを人手で経由しなくてもよい）。

  ざっくりフロー:
    1. ソース projectId + collectionPath
    2. 宛先 projectId + collectionPath（同名 or リネーム）
    3. オプション（サブコレクション含むか、衝突方針）
    4. プレビュー（ソース件数・宛先の既存件数）
    5. 実行 → レポート

  実装イメージ:
    - 内部的には「ソース読取 →（任意で一時）→ 宛先書込」
    - UI 上は transfer 専用。ユーザーに raw JSON を触らせないのが主目的
    - ソース＝宛先の同一 projectId は拒否

  001_002 の Saved Data Transfers:
    転送設定（ソース/宛先/path/オプション）の名前付き保存。
    Phase 11 初期は必須にしない。必要なら 11 後半で saved_queries と同様に
    config/ 配下へ（workspaces.json には入れない）。

  未策定:
    □ 単位: コレクションのみ / ドキュメント複数選択 / クエリ結果転送
    □ サブコレクションを含めるデフォルト
    □ 衝突方針（import と共通化するのがよい）
    □ 宛先が production / 非 read-only の確認ステップ
    □ 双方向同期（diff）は対象外でよいか → v1 は単方向コピー推奨
    □ 失敗時の部分ロールバックはしない（レポートのみ）でよいか


--------------------------------------------------------------------------------
4. Emulator 同期
--------------------------------------------------------------------------------

  やること:
    Emulator と実プロジェクト（または別 Emulator）のあいだでデータを動かす。

  依存:
    - 「つながる」は Phase 12（connection 拡張: ホスト指定・接続方式 UI）
    - Phase 11 の本項目は、接続できたあとの「データを移す」側

  位置づけ（推奨）:
    - 技術的には「3. プロジェクト間転送」のソース/宛先の一方が Emulator のケース
    - 専用 UI を厚くするか、転送 UI に Emulator project が並ぶだけで足りるかは未定
    - Phase 12 前に単独実装すると空振りしやすい → Phase 12 後、または 3 に吸収

  未策定:
    □ Phase 11 Done の「Emulator 向け import/export」を
       A: 転送の特殊ケースとして吸収する
       B: Emulator 専用の「seed / export」ウィザードにする
       のどちらにするか
    □ Firebase の公式エミュレータ import/export 形式との互換を目指すか
       （FireMint 独自アーカイブでよいか）

================================================================================
安全ルール（Phase 11 全体で共通）
================================================================================

  - すべての書込は ensureWritable(projectId)（read-only なら拒否）
  - 宛先が production（environment）のときは確認ダイアログを必須にする方針を推奨
  - プレビューなしの一括書込はしない（少なくとも件数表示）
  - 破壊的オプション（上書き・全削除して入れ直し等）はデフォルト Off
  - IPC は main 経由のみ。renderer がサービスアカウントや生キーに触らない
