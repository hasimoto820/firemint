FireMint 用語（統一・ここが正）

  操作名に「バックアップ」は使わない。
  差は動詞ではなく、対象（コレクション / プロジェクト）で示す。

  | 操作名（UI）                 | 対象                         | ファイル |
  |------------------------------|------------------------------|----------|
  | コレクションをエクスポート   | コレクション 1 つ            | .json    |
  | コレクションにインポート     | コレクション 1 つ            | .json    |
  | プロジェクトをエクスポート   | プロジェクト内の選んだルート | .zip     |
  | プロジェクトにインポート     | プロジェクト（検証→実行）    | .zip     |
  | プロジェクト間でコピー       | プロジェクト A → B（Phase 17）| （内部） |

  補足:
    - 「クエリ結果をエクスポート」… クエリで取ったドキュメント → .json / .csv（Phase 5）
    - サブコレクションを含むかは、各エクスポート／インポートのオプション
    - コード上の関数名・ファイル名に backup が残っても、UI / 仕様文は上表に従う

機能定義
================================================================================

--------------------------------------------------------------------------------
1. コレクションにインポート（JSON → コレクション）
--------------------------------------------------------------------------------

  やること:
    ローカル JSON を、いま接続中の projectId の指定コレクションへ書き込む。

  フロー（決定・プロジェクトにインポートと同型）:
    1. File → コレクションにインポート…
    2. JSON 選択 + サブコレ 含む/除外
    3. 【検証】書込なし。形式・件数・衝突を報告
    4. 検証 OK のときだけ 【インポート実行】が可能
    5. 実行直前にもう一度衝突チェック → batch.create（500）
       … 衝突があれば 1 件も書かない

  想定入力（コレクションをエクスポートした JSON との往復を最優先）:
    JSON:
      ExportDocument[] 相当
        { "id": "...", "path": "...", "data": { ... } }
      ※ path はヒント。id があるときは id を正（親 path + id）

  決定:
    □ ドキュメント ID
        →　ファイルに id がある → その id を使う
           ファイルに id が無い → 自動 ID（firestore 本家 SDK）
           （特定フィールドを ID にする、は v1 ではやらない）

    □ 衝突時
        →　エラー停止。検証時点で検出。書込ゼロ

    □ 検証と実行
        →　独立。検証は dry-run（Firestore への書込なし）

    □ 型
        →　JSON のみ。型もそのまま保存（__firemint_type）。CSV は対象外

    □ サブコレクション
        →　除外/含む を選択式

    □ read-only
        →　検証は可。実行は ensureWritable で拒否

    □ detectEnvironment
        →　使わない（廃止方針）


--------------------------------------------------------------------------------
2. プロジェクトをエクスポート（選んだルート → zip）
--------------------------------------------------------------------------------

  やること:
    指定 projectId のルートコレクション（複数選択可）をたどり、
    ドキュメント（＋方針次第でサブコレクション）を 1 つの zip として保存する。

  コレクションをエクスポート（JSON）との違い:
    - 対象 = プロジェクト内の選んだルートコレクション（複数可）
    - 形式 = zip（manifest + documents）
    - 用途 = プロジェクト単位のまとめ出し

  アーカイブレイアウト（決定）:
    flat + path（コレクションをエクスポート・サブコレ含む、と同じ考え方）
    - 階層フォルダ（collections/users/...）にはしない
    - 各ドキュメントは ExportDocument 相当（id / path / data）。path はフルパス

    export-project-<projectId>-<timestamp>.zip
      manifest.json      … version, projectId, createdAt, 件数サマリ,
                           includeSubcollections, 対象ルート一覧 など
      documents.json     … ExportDocument[]（flat。サブコレ込み時も path で区別）

  決定:
    □ サブコレクション
        →　選択式。「含む／含まない」チェック

    □ 対象ルート
        →　選択式。デフォルトで全ルートにチェック

    □ 圧縮形式
        →　zip 固定

    □ 巨大時
        →　500 件読んでファイルに追記／分割書き。RAM に溜めない

    □ 進捗 UI
        →　件数 / コレクション名 / 進捗率

    □ プロジェクトにインポート（旧 restore）
        →　11-c（本節の次）で実施。検証と実行を分ける


--------------------------------------------------------------------------------
3. プロジェクトにインポート（zip → フォーカス中のプロジェクト）
--------------------------------------------------------------------------------

  やること:
    プロジェクトをエクスポートした zip（manifest.json + documents.json）を、
    いまフォーカス中の projectId へ書き戻す。

  コレクションにインポートとの違い:
    - 対象 = プロジェクト全体（path 付き flat ドキュメント）
    - 形式 = zip
    - 衝突チェックと書込を分ける（検証独立）

  フロー（決定）:
    1. File → プロジェクトにインポート…
    2. zip 選択
    3. 【検証】書込なし。形式・件数・衝突・manifest.projectId 不一致を報告
    4. 検証 OK のときだけ 【インポート実行】が可能
    5. 実行直前にもう一度衝突チェック → batch.create（500）
       … 衝突があれば 1 件も書かない（コレクションにインポートと同方針）

  決定:
    □ 衝突
        →　エラー停止。検証時点で検出。書込ゼロ

    □ 検証と実行
        →　独立。検証は dry-run（Firestore への書込なし）

    □ 宛先
        →　フォーカス中の projectId 固定

    □ manifest.projectId 不一致
        →　警告表示。ユーザーが確認してから実行可

    □ サブコレクション
        →　ファイルに入っている path をそのまま書く
           （エクスポート時に選んだ結果が zip に入っている）

    □ read-only
        →　検証は可。実行は ensureWritable で拒否

    □ 巨大時（v1）
        →　documents.json は一括 parse（コレクション import と同程度）
           ストリーム化は後続改善

================================================================================
安全ルール（Phase 11 全体で共通）
================================================================================

  - すべての書込は ensureWritable(projectId)（read-only なら拒否）
  - プレビューなしの一括書込はしない（少なくとも件数表示）
  - 破壊的オプション（上書き・全削除して入れ直し等）はデフォルト Off
  - IPC は main 経由のみ。renderer がサービスアカウントや生キーに触らない
  - 大きな読み出しは、読みながらディスクへ（500 件ページ）。RAM に溜めない
